# Инструкция для Backend Developer (Rust)

## Проблема
На фронтенде отображаются некорректные данные в скринере:
1. **OI Δ 8H** (Изменение открытого интереса за 8 часов) = `0.00`
2. **CHANGE 1D** (Изменение цены за 24 часа) = `0.00%`
3. **OPEN INTEREST** (Открытый интерес в $) = `0.00`
4. **DAY** (Изменение с начала дня UTC) = `+Infinity%`
5. **WEEK** (Изменение с начала недели) = `+Infinity%`

Это указывает на то, что:
- Данные либо не собираются, либо не сохраняются.
- При расчетах происходит деление на ноль (Infinity), что говорит об отсутствии исторических данных (цена открытия дня/недели = 0).

## Задача
Необходимо реализовать корректный сбор, хранение и отдачу данных для следующих полей.

### 1. Open Interest (OI) и OI Change
**Источник:** Binance Futures API (`GET /fapi/v1/openInterest`).
**Логика:**
- Периодически (например, раз в 1-5 минут) опрашивать OI для всех тикеров.
- Хранить историю OI минимум за последние 24 часа (в памяти или БД).
- **Расчет `openInterestValue`**: Текущее значение OI в USDT (обычно `openInterest * markPrice`).
- **Расчет `oiChange8h`**: 
  ```rust
  let oi_now = current_oi;
  let oi_8h_ago = get_oi_at(now - 8h);
  let change = (oi_now - oi_8h_ago) / oi_8h_ago * 100.0;
  ```
  *Если `oi_8h_ago` отсутствует, возвращать `null` или `0`, но не `Infinity`.*

### 2. 24h Change (Change 1D)
**Источник:** Binance Futures API (`GET /fapi/v1/ticker/24hr`).
**Поле:** `priceChangePercent`.
**Логика:**
- Это значение отдает сама биржа. Нужно просто корректно прокидывать его в поле `priceChangePercent24h` или `change1d` в структуре `ScreenerRow`.
- Убедиться, что парсинг JSON корректен (строка -> число).

### 3. Day Change & Week Change
**Источник:** Исторические свечи (Klines).
**Логика:**
- **Day Open:** Цена открытия свечи `1d` (UTC 00:00).
  - Нужно хранить `dayOpen` цену для каждого символа.
  - Обновлять раз в сутки или при старте сервера подгружать через `GET /fapi/v1/klines?interval=1d&limit=1`.
  - Расчет: `(currentPrice - dayOpen) / dayOpen * 100.0`.
- **Week Open:** Цена открытия свечи `1w` (Понедельник UTC 00:00).
  - Аналогично, подгружать через `GET /fapi/v1/klines?interval=1w&limit=1`.
  - Расчет: `(currentPrice - weekOpen) / weekOpen * 100.0`.

**Важно:** Если `dayOpen` или `weekOpen` равны 0 (или данные не загрузились), не отдавать `Infinity`. Отдавать `null` или `0`.

## Структура ответа (ScreenerRow)
Убедитесь, что JSON ответ содержит заполненные поля:

```json
{
  "symbol": "BTCUSDT",
  "price": 95000.0,
  "openInterestValue": 5000000000.0,  // Исправить 0.00
  "oiChange8h": 1.5,                  // Исправить 0.00
  "change1d": 2.4,                    // Исправить 0.00%
  "dayOpen": 94000.0,                 // Нужно для расчета DAY %
  "weekOpen": 90000.0,                // Нужно для расчета WEEK %
  ...
}
```

## Резюме действий
1. Настроить фоновый воркер для сбора OI (каждые 5 мин).
2. Настроить кэширование цен открытия дня (`dayOpen`) и недели (`weekOpen`).
3. Исправить формулы расчета процентов (проверка на деление на ноль).
4. Проверить маппинг полей из Binance API в структуру ответа API.
